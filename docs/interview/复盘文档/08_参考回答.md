
### 业务理解类

**Q1：为什么选择做"治理"而不是做一个更通用的知识问答 chatbot？**

> 因为我真正想解决的问题不是"问一个问题得到答案"，而是"笔记库越来越乱怎么办"。问答只是消费已有知识，治理是要发现笔记里的问题——比如标签缺失、内容过时、关联断裂——然后帮你修。这两件事的技术链路也不一样：chatbot 只需要检索加生成，治理还需要分析、提案、审批、写回、审计。所以我从一开始就把它定位成一个治理系统。

**Q2：你定义的"治理"具体包含哪些操作？边界在哪？**

> 当前实现的治理操作收敛在两类 patch op 上：一个是 merge_frontmatter——把 frontmatter 里缺失的字段补上，比如 tags、aliases；另一个是 insert_under_heading——在某个标题下插入内容。我有意不做全文覆写或者删除操作，因为出错代价太高。边界就是"只做加法和补充，不做删除和重排"。

**Q3：Obsidian 笔记有什么特殊性？**

> 三个特点。第一，它是 Markdown，有 frontmatter、heading 层级结构、wikilink 互联，不是纯文本。第二，笔记之间通过 wikilink 形成图结构，治理的时候需要考虑关联。第三，用户对自己的笔记有很强的控制欲——不像 wiki 那样大家都能改，个人笔记被 AI 乱改是不能接受的。这就是为什么必须有审批。

**Q4：为什么治理操作必须经过人工审批？**

> 因为 LLM 有幻觉。比如它分析出某篇笔记应该加一个标签，但这个判断可能是错的。如果直接写回，用户打开笔记发现多了一堆奇怪的标签，信任就没了。所以我设计成 proposal 模式——AI 说"我建议做这些改动"，用户看 diff 预览，觉得合理才批准。这个信任链路比技术实现更重要。

**Q5：目标用户是谁？**

> 目前是面向个人用户的，就是重度 Obsidian 使用者。没有做多用户或团队版。这是一个个人项目，没有上线生产环境。

### Agent 架构类

**Q6：workflow-first 跟 ReAct 具体区别是什么？**

> ReAct 是让模型自己决定"思考 → 行动 → 观察"的循环，循环几次、调什么工具都由模型控制。这在开放式问答场景很灵活，但在治理场景有两个问题：一是不确定性太强，模型可能循环很多次不收敛；二是治理链路是固定的——分析、提案、审批、写回——这些步骤的顺序不能变，也不应该让模型决定跳过某一步。
> 
> 所以我选了 workflow-first：每条工作流是一个固定的节点序列，用 graph runner 按顺序执行。模型在特定节点上被调用做分析和生成，但"下一步做什么"是编排层决定的，不是模型决定的。

**Q7：为什么自研 graph runner 而不是直接用 LangGraph？**

> 两个原因。一是我的场景只需要线性图——节点 A 执行完到 B 再到 C，不需要条件分支或并行。LangGraph 的完整功能对我来说过重了。二是我想减少运行时依赖——LangGraph 安装会带进一大堆依赖包。所以我自己写了一个轻量 shim，接口兼容 LangGraph 的 StateGraph，当 LangGraph 装了就用原生的，没装就用我的 shim。这样未来如果需要更复杂的图编排，切换成本很低。

**Q8：兼容 shim 具体兼容到什么程度？**

> 坦白说只兼容了线性图。我的 shim 支持 `add_node`、`add_edge`、`compile`、`invoke`，足够串联线性节点。条件分支（`add_conditional_edges`）和并行执行不支持。这在当前场景够用，但如果未来要做 DAG 编排，就需要升级到 LangGraph 原生或者重写 runner。
>
> 补充说一下 shim 的实现机制：`runtime.py` 里用 `try: from langgraph.graph import StateGraph` / `except ImportError` 做条件导入。如果 LangGraph 已安装就直接用原生实现，没装就走自研的 `StateGraph` 类。这样业务代码（ask_graph.py / ingest_graph.py / digest_graph.py）完全不需要知道底层跑的是哪个实现，调用方式完全一致。`WorkflowGraphExecution` 里有一个 `used_langgraph` 字段记录当次执行用的是哪个 runner，方便 trace 和排查。

**Q8.1：LangGraph 有 `interrupt()` 做人工审批中断，你的项目也需要审批，为什么不直接用？**

> 这个问题要结合 Q7 一起看。核心原因不是技术上做不到——LangGraph 自己也有持久化 checkpointer（比如 `SqliteSaver`），也能跨请求恢复，也支持多 thread 并发等审批。真正的原因是**依赖取舍**：既然我已经选择了自研 shim 来避免 LangGraph 的运行时依赖，审批中断作为 checkpoint 状态机的自然延伸就顺手实现了，没必要为了一个 `interrupt()` 特性反过来引入完整的 LangGraph。
>
> 具体方案是**两阶段调用 + checkpoint 持久化**：第一阶段，工作流执行到审批点，把 proposal 和 checkpoint（状态设为 `WAITING_FOR_APPROVAL`，`next_node_name="human_approval"`）**原子写入 SQLite**，然后请求直接结束返回。第二阶段，用户在 Obsidian 插件里审批后，前端调用 `/workflows/resume`，后端从 SQLite 加载 checkpoint 恢复状态，继续执行后续节点（writeback、audit 等）。
>
> checkpoint 按 `(thread_id, graph_name)` 隔离，所以**支持多个工作流同时等审批**——比如同时对 5 篇笔记发起 ingest，就有 5 个 thread 各自处于 `WAITING_FOR_APPROVAL`，互不干扰。
>
> 关键的工程细节是 proposal 和 waiting checkpoint 必须原子落盘——用 SQLite 事务保证。否则可能出现插件拿到了 proposal 但 thread 状态还在 `in_progress`，导致 pending inbox 查询和 resume 失配。

**Q9：checkpoint 存 SQLite，多实例怎么办？**

> 确实不支持。SQLite 是单文件单写者模型，多实例并发写入会冲突。但这是个人项目，单实例完全够用。如果要做多实例，checkpoint 需要迁移到 PostgreSQL 或者 Redis，然后加分布式锁。这个我目前没做，因为不是当前的需求。

**Q10：三条工作流之间有没有共享状态？**

> 有一个间接共享：ingest 写回成功后会触发 scoped re-index，更新 SQLite 的 note/chunk 索引，ask 下次查询时就能检索到最新内容。但三条工作流的 checkpoint 是按 thread_id 隔离的，不会互相干扰。状态共享只通过索引数据库这个公共存储层发生。

**Q11：加新工作流改动量有多大？**

> 实际上不大。handler registry 模式下，加一条新 workflow 需要三步：一是写 graph 文件定义节点序列，二是写 service 层实现业务逻辑，三是在 main.py 的 `WORKFLOW_INVOKE_HANDLERS` 字典里注册 invoker 和 outcome builder。contract 层的 WorkflowAction 枚举加一个值就行。这是我做统一 workflow contract 重构的目标之一。

### Prompt / Workflow 类

**Q12：prompt 模板怎么管理的？**

> prompt 模板在 `backend/app/context/render.py` 里，以 Python 函数的形式管理——比如 `render_grounded_answer_prompt()`、`render_tool_selection_prompt()`。没有用独立的 prompt 文件或模板引擎。版本控制就是代码 git 版本。坦白说这块没有做到很精细的 prompt 版本管理，如果要做 A/B test 或者 prompt 回滚，当前架构需要改进。

**Q13：ask 链路里模型被调用了几次？**

> 正常路径最多两次。第一次是 tool selection——模型看完 context bundle 后决定要不要调工具、调哪个工具。第二次是 grounded answer generation——根据检索结果和（可选的）工具执行结果生成带引用的回答。如果第一次判断不需要工具，就跳过工具执行直接进第二次。如果两次都失败了就降级到 retrieval_only。

**Q14：模型不按格式返回怎么办？**

> 目前主要靠两个机制。一是 tool call 的 JSON Schema 运行时校验——如果模型返回的工具参数不合法，guardrail 会拦截并降级。二是 citation alignment 校验——如果模型生成的答案里引用编号（[1] [2]）跟实际候选集不对齐，就降级到只返回检索片段。但要承认，对于模型生成的自然语言文本格式，目前没有很强的结构化约束，主要靠 prompt 引导。

**Q15：ingest 的 proposal 是模型生成的还是规则生成的？**

> 分析阶段是模型做的——分析一篇笔记需要什么治理操作。但 proposal 的结构是规则化的——分析结果会被转成结构化的 PatchOp 列表，每个 PatchOp 有固定的 op 类型、target_path 和 payload。所以是"模型做语义判断，规则做结构化封装"。

**Q16：digest 复盘内容怎么生成的？**

> digest 会查最近一段时间的笔记变更，用模型生成摘要。有 proposal 模式——生成的摘要可以作为 proposal 提交审批后写入复盘笔记。具体的去重逻辑——待补充，我需要确认 digest 是否有跨周期去重。

### 检索与 RAG 类

**Q17：RRF 的 k=60 怎么确定的？**

> 说实话这是一个经验值。RRF 原论文推荐的 k 一般在 60 左右，我直接用了这个值。没有做超参搜索或对比实验。如果要严格调优，应该在有标注的检索测试集上跑 grid search 比较不同 k 值对 MRR 的影响。这是一个待改进的点。

**Q18：FTS5 对中文支持怎样？**

> SQLite FTS5 原生分词器对中文支持有限，它按空格分词，中文连续字符会被当作一个 token。待补充——我需要确认当前是否引入了额外的中文分词器（如 jieba 分词后再建索引），还是依赖 FTS5 的默认行为。

**Q19：向量检索用的什么 embedding 模型？**

> embedding 模型是通过配置指定的——Settings 里有 `cloud_embedding_model` 和 `local_embedding_model` 两个配置项，具体用的哪个模型取决于运行时配置。维度——待补充，需要确认 `chunk_embedding` 表的列定义。

**Q20：检索结果最多返回几条？为什么？**

> ask 链路 `ASK_RETRIEVAL_LIMIT=4`。选 4 是一个平衡：太少可能漏掉相关内容，太多会占用 token 预算（只有 2400），而且长上下文会分散模型注意力。这个值没有做严格的对比实验，是一个工程判断。

**Q21：token 预算 2400 怎么定的？**

> 也是工程判断。考虑的是：留足够空间给模型生成回答（通常几百 token），加上 system prompt 和 tool description 的开销，2400 作为检索上下文部分的预算在常见 4K-8K 的 context window 下是合理的。但这个值是硬编码的，没有根据具体模型的 context window 动态调整。

**Q22：笔记库扩大 10 倍怎样？**

> FTS5 查询性能应该还行，SQLite FTS5 对中等规模（万级文档）的全文搜索是够用的。瓶颈更可能在向量检索——sqlite_vector 是暴力扫描还是有索引取决于实现。另一个瓶颈是 ingest 的整库 FTS 重建——当前每次 scoped ingest 都会重建 chunk_fts，如果 approval 频繁触发会很慢。这正是 TASK-032 要解决的问题：收敛成局部 FTS 同步。

### Tool Calling 类

**Q23：模型绕过白名单会怎样？**

> `validate_tool_call()` 会检查工具名是否在 `TOOL_REGISTRY` 里。如果不在，返回 `DOWNGRADE_TO_RETRIEVAL_ONLY` 的 guardrail outcome，ask 链路就降级到只返回检索结果，不执行任何工具。所以不存在"绕过"——不在白名单里的工具名根本无法执行。

**Q24：为什么全部 read-only？**

> 这是有意的设计限制。ask 工作流里调用工具的目的是"补充检索信息"——比如加载某篇笔记的完整内容——不需要写操作。治理场景的写操作走的是 proposal → 审批 → writeback 这条独立链路，不经过 tool calling。如果未来要加写操作工具，我会在 ToolSpec 里加 `write_scope` 字段，并且要求写操作工具只能在 ingest/digest 工作流里使用，同时加更严格的参数校验和审计。

**Q25：Schema 校验具体验什么？**

> `_arguments_match_schema()` 做的是标准 JSON Schema 验证。比如 search_notes 工具要求 query 是 string 类型，limit 是 1-20 的 integer。如果模型输出的 arguments 里 query 不是 string，或者 limit 是 -1，校验会失败，guardrail 拦截。有没有遇到过——在测试中人为构造过不合法参数的 case，guardrail 确实拦住了。生产环境没有数据，因为没上线。

**Q26：工具执行超时怎么处理？**

> 模型调用有 `MODEL_TIMEOUT_SECONDS=20` 的超时控制。工具执行本身是本地 SQLite 查询，通常毫秒级，暂时没有显式超时。如果工具执行抛异常，`execute_tool_call()` 会返回 `ok=False` 的 `ToolExecutionResult`，下游 guardrail 根据 result 判断是否降级。

### 稳定性与容错类

**Q27：cloud 和 local 模型能力差异怎么处理？**

> 确实有差异，cloud 通常用更强的模型（如 GPT-4 级别），local 可能是更小的模型。降级到 local 时回答质量可能下降。当前没有做特殊处理——两个 provider 用同样的 prompt。如果要精细化，可以针对不同模型调整 prompt 或 token 预算。但我的设计哲学是"宁可降级返回一个粗糙的结果，也不返回 500 错误"。

**Q28：citation alignment 具体检查什么？**

> 我在模型返回的答案里用正则 `\[(\d+)\]` 提取所有引用编号，然后检查两件事：一是编号不能缺失（有回答但没有任何引用不行）；二是编号不能越界（比如只有 3 个候选片段，但答案里出现了 [4] 或 [5]）。如果不对齐，就降级到 retrieval_only，只返回原始检索片段让用户自己看。

**Q29：降级后用户体验怎样？**

> 降级到 retrieval_only 时，用户看到的不是"服务不可用"，而是"我找到了这些相关的笔记片段，但无法生成总结性回答"。每个片段带有来源路径、标题、heading path 和 snippet。用户可以自己点进去看原文。体验比生成回答差，但比空白页好。

**Q30：trace 丢数据你怎么知道？**

> 坦白说，当前没有专门的 trace 完整性监控。trace hook 异常被吞掉后只是 `continue`，没有写日志也没有计数器。如果要改进，应该加一个失败计数器或者 fallback 告警。但当前这个设计的优先级判断是：trace 是观测补充，不能因为 trace 写失败把主链路搞挂。

### 效果评测类

**Q31：21 条 case 够吗？**

> 不够，但是一个可用的起点。21 条覆盖了主路径——正常问答、无结果、需要工具、guardrail 拦截、治理 proposal 生成、审批恢复、digest 等。边界 case 和对抗 case 不足。如果要提升，需要：(1) 根据线上数据（如果有的话）补充真实 case；(2) 加模糊查询、多义词、跨语言等边界 case；(3) 加 prompt injection 对抗 case。

**Q32：没有语义级评测吗？**

> 当前只有确定性断言，没有 LLM-as-judge 或者人工标注的语义评分。确定性断言能检查"结构是否正确"（比如 citations 非空、mode 是否符合预期），但不能评估"回答质量好不好"。如果要做语义评测，我会考虑两个方向：一是用 LLM-as-judge 打分，但需要注意评分标准的一致性；二是人工标注一批参考回答做相似度比较。这是一个明确的待改进项。

**Q33：eval 有没有接 CI？**

> 没有。当前是手动运行 `python eval/run_eval.py`。没有接 CI 的原因是 eval 需要模型调用，CI 环境需要配置 API key 和网络访问。如果要接 CI，可以分两层：确定性断言部分（不需要模型的 case）可以直接跑，需要模型的 case 走 scheduled CI 或者用 mock。

**Q34：怎么评估检索召回率？**

> 当前没有独立的检索 benchmark。如果要做，我会：(1) 标注一批 query-relevant_chunks 对；(2) 对每个 query 跑 hybrid 检索，计算 recall@4（因为 limit=4）和 MRR；(3) 对比纯 FTS、纯向量、RRF 三种方案的差异。这个数据当前没有，是一个明确的 gap。

### 成本与性能类

**Q35：单次 ask 请求调几次模型？**

> 正常路径最多两次：一次 tool selection，一次 grounded answer generation。如果不需要工具就一次。token 消耗——待补充，当前没有 token 计数统计。粗估：检索上下文 2400 字符 + system prompt + query + 生成回答，每次模型调用大约 2000-4000 token，两次大约 4000-8000 token。

**Q36：做过延迟优化吗？**

> 没有做过专门的延迟优化。当前瓶颈大概率在模型调用——两次 HTTP 请求到 LLM 服务，每次 1-5 秒不等。检索和装配是本地 SQLite 操作，通常毫秒级。如果要优化，首先可以考虑：(1) tool selection 和 answer generation 能否合并成一次调用；(2) 能否加响应缓存；(3) 能否用更快的模型做 tool selection。

**Q37：怎么降低成本？**

> 几个方向：(1) 对高频重复查询加缓存——当前没做；(2) tool selection 用小模型而不是大模型——当前两次调用用同一个 provider；(3) 缩减 context 预算——但会影响回答质量，是 trade-off；(4) 对不需要模型的查询做规则级 short-circuit——比如纯关键词查询直接返回 FTS 结果。

**Q38：SQLite 单库有没有瓶颈？**

> 当前规模下没有。SQLite 对于读多写少的场景性能很好，个人笔记库几百到几千篇文档完全没问题。瓶颈会出现在：(1) 高并发写入——SQLite 单写者模型；(2) ingest 整库重建 chunk_fts 时会锁表——这个已经识别为 TASK-032 的问题；(3) trace 数据积累很快，如果不清理会导致数据库膨胀。

### 工程化与可维护性类

**Q39：为什么用 urllib 而不是 requests？**

> 减少依赖。urllib 是 Python 标准库，不需要额外安装。模型调用就是一个 HTTP POST + JSON body，urllib 完全够用。如果要做连接池、重试、async 这些高级功能，再换 httpx。当前场景不需要。

**Q40：为什么用 SQLite 而不是 PostgreSQL？**

> 三个原因。一是部署简单——SQLite 是单文件，不需要起数据库服务。二是个人项目规模小，不需要 PostgreSQL 的并发和分布式能力。三是 Obsidian 本身就是本地应用，数据放本地更自然。如果要上云或者支持多用户，就需要迁移到 PostgreSQL。

**Q41：104 条测试覆盖什么？**

> 包括单元测试和集成测试。单元测试覆盖：parser 解析、ingest 幂等写入、checkpoint CRUD、proposal/audit 持久化、tool 校验、guardrail 判断等。集成测试覆盖：invoke 端到端（ask/ingest/digest）、resume 审批恢复端到端。不包含插件端测试（TS 测试是单独的）。覆盖率数字——待补充。

**Q42：中文注释是有意的吗？**

> 是的。这个项目本来就是给自己用的，中文注释帮助我自己理解和维护。代码里的关键决策注释（比如为什么 filter fallback 放在 hybrid 入口）用中文写是因为这些注释更像是设计 memo，中文更准确。如果是开源项目或团队项目，注释和文档应该统一用英文。
