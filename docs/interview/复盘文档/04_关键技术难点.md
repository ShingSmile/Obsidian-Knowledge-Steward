
### 难点 1：有状态工作流的编排与持久化

**问题**：治理类操作（ingest proposal、digest proposal）需要人工审批，工作流不能同步阻塞等待，必须支持"挂起 → 恢复 → 继续执行"。

**原因分析**：人工审批时间不确定（秒级到天级），HTTP 请求不能一直挂着；且审批可能跨会话，需要持久化中间状态。

**方案**：自研 checkpointed linear graph runner：

- 每个 graph node 执行完后 checkpoint 写入 SQLite `workflow_checkpoint` 表
- 审批类节点产出 proposal 后进入 `waiting_for_approval` 状态并挂起
- `/workflows/resume` 接收审批决策，加载 checkpoint，从断点继续执行
- thread_id 贯穿整个生命周期，支持按 thread 回溯

**落地实现**：

- [runtime.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/graphs/runtime.py) — `invoke_checkpointed_linear_graph()`，支持 `save_graph_checkpoint` / `load_graph_checkpoint`
- [checkpoint.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/graphs/checkpoint.py) — SQLite checkpoint 持久化
- 当 LangGraph 可用时使用其 StateGraph，不可用时 fallback 到自研 `_CompiledStateGraph` shim

**效果**：三条工作流（ask/ingest/digest）统一使用同一 graph runner，审批链路可跨会话恢复。104 条后端测试 + 21 条离线 eval 覆盖主路径。

**风险与权衡**：

- 自研 shim 只覆盖线性图，不支持条件分支/并行——当前治理场景不需要，但如果未来需要 DAG 编排则需重新评估
- checkpoint 存 SQLite 单库，不支持分布式——个人项目规模下足够，但不适合多实例部署

### 难点 2：Human-in-the-Loop 安全写回

**问题**：Agent 对用户笔记的修改必须可信、可追溯、可回滚，不能出现"AI 悄悄改了我的笔记"的情况。

**原因分析**：LLM 有幻觉风险，直接写回会破坏用户数据信任。知识治理类场景用户对准确性要求极高。

**方案**：proposal → 审批 → 受控写回 → 审计的完整链路：

1. 后端生成结构化 proposal（包含 patch_op 列表、evidence、risk_level、safety_checks）
2. Checkpoint 挂起，proposal 持久化到 SQLite `proposal` 表
3. 插件端拉取 proposal，展示 diff 预览
4. 用户 approve/reject + 填写 reviewer/comment
5. 插件执行受限 patch op（仅 `merge_frontmatter` / `insert_under_heading`）
6. 写回前校验 `before_hash`，写回后生成 `after_hash`
7. WritebackResult 回传后端，写入 append-only `audit_log`

**落地实现**：

- [contracts/workflow.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/contracts/workflow.py) — `Proposal`, `PatchOp`, `SafetyChecks`, `AuditEvent` 数据模型
- [resume_workflow.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/services/resume_workflow.py) — 审批恢复 + 状态校验
- [applyProposalWriteback.ts](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/plugin/src/writeback/applyProposalWriteback.ts) — 插件端受限写回
- [store.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/indexing/store.py) — `proposal`, `audit_log` 表 + append-only 约束

**效果**：每次写回都有完整审计链路，before_hash/after_hash 保证文件完整性。

**风险与权衡**：

- 当前只支持两类 patch op，复杂编辑（如删除段落、重排结构）尚不支持——这是有意限制，降低出错概率
- rollback 是 patch-level，不是 snapshot-level——对于 merge_frontmatter 这类操作够用，但全文重写场景需要升级
- **已知 gap**：本地写回成功但 `/workflows/resume` 失败时，缺乏跨会话恢复入口（对应 TASK-031，待补充）

### 难点 3：混合检索管线与 RRF 融合

**问题**：单路检索（纯 FTS 或纯向量）在不同查询类型上各有短板——关键词查询向量弱，语义查询 FTS 弱。

**原因分析**：FTS5 基于 BM25 排序，擅长精确关键词匹配；向量检索擅长语义相似度但对精确词匹配不敏感。用户查询两种都有。

**方案**：双路检索 + Reciprocal Rank Fusion（RRF）融合：

- FTS5 分支和 vector 分支各自独立检索
- 通过 `_HybridCandidateAccumulator` 按 chunk_id 累加 RRF 分数：`score += 1 / (k + rank)`，k=60
- 融合后按 fused_score 降序取 top-k
- filter 放在 hybrid 总入口统一处理，避免两条分支 filter 漂移

**落地实现**：

- [hybrid.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/retrieval/hybrid.py) — `search_hybrid_chunks()`，`HYBRID_RRF_K=60`，`HYBRID_BRANCH_MIN_LIMIT=8`
- filter fallback：主查询带 filter 无结果时，在 hybrid 入口统一去掉 filter 重试，而不是让每条分支各自放宽

**效果**：待补充——当前没有独立的检索 benchmark 数据（如 recall@k、MRR 对比），只在 ask eval 的 21 条 golden case 中间接验证。

**风险与权衡**：

- RRF k=60 是经验值，未做超参搜索——待补充对比实验
- FTS5 在中文场景依赖分词质量——待补充分词器配置细节
- 向量分支的 embedding 模型选型、维度、索引方式——待补充具体参数

### 难点 4：上下文装配控制面

**问题**：检索结果不能直接塞进 prompt——可能有重复、超出 token 预算、含有不安全内容。

**原因分析**：不控制上下文规模会导致：(1) token 成本失控；(2) 长上下文干扰模型注意力，降低回答质量；(3) 不安全内容可能触发 prompt injection。

**方案**：在检索和模型调用之间插入独立的 context assembly 控制面：

1. `_dedupe_candidates()` — 按 chunk_id 去重
2. `_trim_candidates_to_budget()` — 按字符估算裁剪到 token_budget（当前 2400）
3. `_collect_flags()` + `detect_safety_flags()` — 安全标记检测
4. 打包为 `ContextBundle`，附带 `assembly_notes` 记录处理过程（如 "deduplicated:2", "trimmed_for_budget:1"）

**落地实现**：

- [assembly.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/context/assembly.py) — 三个 builder：`build_ask_context_bundle`, `build_ingest_context_bundle`, `build_digest_context_bundle`
- 不同 workflow 的装配逻辑不同：ask 以检索候选为主，ingest 以 proposal evidence 为主

**效果**：通过 assembly_notes 可追溯每次装配的去重数、裁剪数、safety flag，便于调试。

**风险与权衡**：

- token 预算 2400 是硬编码常量，未做动态调整——当前够用，但长文档场景可能需要自适应
- 字符估算而非真实 tokenizer 计数——存在 ±20% 误差，但避免了 tokenizer 依赖

### 难点 5：受控 Tool Calling + 纵深防御

**问题**：让模型决定调用哪个工具天然存在风险——模型可能被 prompt injection 诱导调用危险工具或传入恶意参数。

**原因分析**：开放式 ReAct 循环中，工具调用完全由模型控制，攻击面大。治理场景对安全性要求高，不能容忍工具滥用。

**方案**：5 层防御纵深：

1. **白名单注册表**：`TOOL_REGISTRY` 只有 3 个工具（search_notes, load_note_excerpt, list_pending_approvals）
2. **Workflow 隔离**：每个工具声明 `allowed_workflows`，ask 工具不能在 ingest 里用
3. **Read-only 强制**：所有工具 `read_only=True`，从定义层面禁止写操作
4. **运行时 Schema 校验**：`_arguments_match_schema()` 对入参做 JSON Schema 验证
5. **Guardrail 拦截**：`evaluate_tool_call_outcome()` 在执行前检查，`evaluate_final_ask_response()` 在生成后检查

**落地实现**：

- [registry.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/tools/registry.py) — `TOOL_REGISTRY` + `validate_tool_call()`
- [guardrails/ask.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/guardrails/ask.py) — 前后置 guardrail
- [context/safety.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/context/safety.py) — safety_flags 检测

**效果**：eval 中包含 `tool_and_guardrail` 场景的 golden case 覆盖。

**风险与权衡**：

- 当前没有红队对抗测试数据验证防御效果——待补充
- 3 个工具全部 read-only，如果未来需要写操作工具，需要重新设计权限模型
- Schema 校验是运行时做的，不是编译时——但考虑到工具参数来自模型输出，运行时校验是合理选择

### 难点 6：多层 Fallback 降级策略

**问题**：主链路中任何一环失败（检索无结果、模型调用超时、引用不对齐、guardrail 拦截）都不应该返回空白或报错。

**原因分析**：模型服务不稳定（超时、配额、网络），检索可能无匹配，LLM 输出不可控——任何一环都可能失败。

**方案**：逐层降级：

- **检索层**：filter 无结果 → 去 filter 重试
- **模型层**：cloud provider 失败 → 切 local provider（`_resolve_chat_provider_targets()`，优先级 cloud=1, local=2）
- **引用校验**：编号不对齐 → 降级为 `retrieval_only` 模式，只返回检索片段不返回生成答案
- **Guardrail**：safety flag 触发 → 降级为 `retrieval_only`
- **工具调用**：guardrail 拦截 → 跳过工具，直接用检索结果生成
- **全链路兜底**：所有降级路径都经 `_build_retrieval_only_result()` 统一返回

**落地实现**：

- [ask.py:60-230](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/services/ask.py#L60-L230) — 完整降级链路，每个分支都有对应的 `_build_retrieval_only_result()` 调用

**效果**：在 eval 中，模型不可用时仍能返回 `retrieval_only` 结果而非 500 错误。

**风险与权衡**：

- 降级意味着回答质量下降，但用户至少能看到检索到的原始片段
- 没有对降级率做持续监控——待补充

### 难点 7：离线评测框架

**问题**：Agent 输出不确定，没有自动化评测就无法判断每次改动是改进还是退化。

**原因分析**：LLM 输出天然不稳定，修改检索逻辑、prompt 模板、工具行为后，需要回归验证。

**方案**：场景化 golden case + 分组 benchmark：

- 21 条 golden case，覆盖 ask、tool_and_guardrail、governance、digest、resume_workflow 五个场景
- 按面试叙事主线拆成两组 benchmark：`question_answering` 和 `governance`
- 每个 case 有确定性断言：检查 mode、citations 非空、proposal 结构、audit_log 写入等
- `RESULT_SCHEMA_VERSION=1.3`，结果输出含 `benchmark_overview`、场景级 `core_metrics`、`failure_type_breakdown`

**落地实现**：

- [eval/run_eval.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/eval/run_eval.py) — eval runner
- [eval/golden/ask_cases.json](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/eval/golden/ask_cases.json) — golden case 定义

**效果**：21/21 case 通过，104 条后端单元/集成测试通过。

**风险与权衡**：

- 21 条 case 覆盖面有限，边界 case 不足——待补充
- 没有 LLM-as-judge 的语义评测，只有确定性断言——对检索相关性、回答质量的评估粒度较粗
- 没有 CI 集成——当前手动运行

### 难点 8：双写 Trace 可观测体系

**问题**：Agent 链路长（检索 → 装配 → 工具 → 生成 → 校验 → guardrail），出问题时很难定位是哪一环。

**原因分析**：Agent 不像传统 API 只有请求-响应，中间有多个"模型决策点"，需要全链路可追溯。

**方案**：JSONL + SQLite 双写 trace：

- `build_jsonl_trace_hook()` — 追加写 JSONL 文件，适合流式查看
- `build_sqlite_trace_hook()` — 写入 `run_trace` 表，适合结构化查询
- `compose_trace_hooks()` — 组合多个 hook，每个 hook 独立 try/catch，一个失败不影响其他
- trace_id 用 SHA1(内容) 生成，SQLite INSERT 用 `ON CONFLICT DO NOTHING` 去重
- 提供 CLI：`python -m app.observability.runtime_trace --thread-id xxx`

**落地实现**：

- [runtime_trace.py](vscode-webview://02oh081k7rv74jnvqfcaic9c2gv5f7g1g8chnggcci2rhg7nfvtv/backend/app/observability/runtime_trace.py) — 完整实现

**效果**：可按 thread_id 回溯一次完整工作流的所有节点事件。

**风险与权衡**：

- trace 只记录事件，不记录 prompt 全文和模型完整输出——是有意为之，避免 trace 文件过大
- 没有接入 OpenTelemetry 或 LangSmith 等标准化方案——个人项目规模下自研够用
