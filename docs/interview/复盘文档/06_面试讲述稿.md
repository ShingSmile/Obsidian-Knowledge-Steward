
### 1 分钟版本

> 我做的是一个基于 LLM 的个人知识治理 Agent 系统。它不是一个简单的问答 chatbot，核心解决的是"用 AI 辅助管理 Obsidian 笔记库"的问题——包括知识问答、文档治理和周期复盘三条工作流。
> 
> 架构上我选择了 workflow-first 的方案而不是 ReAct，因为治理场景需要确定性流程。后端用 Python + FastAPI，自研了一个支持 checkpoint 挂起恢复的 graph runner，检索用 FTS5 + 向量双路 RRF 融合，工具调用有白名单 + 5 层防御。治理操作必须经过人工审批，写回有 hash 校验和审计日志。
> 
> 整个系统我独立设计实现，目前有 104 条后端测试和 21 条场景化 eval case 覆盖。

### 3 分钟版本

> 我做的是一个基于 LLM 的个人知识治理 Agent 系统，用于管理 Obsidian 笔记库。核心定位是治理而不是问答——问答只是其中一条链路，还有文档治理（ingest）和周期复盘（digest）。
> 
> **架构选型**：我选了 workflow-first 而不是 ReAct。原因是治理场景需要确定性——比如 ingest 要走"分析 → 生成 proposal → 等人审批 → 受控写回"这样固定的链路，不能让模型自己决定循环几次。我自研了一个 checkpointed linear graph runner，兼容 LangGraph 的 StateGraph 接口，但不强依赖它。核心能力是 checkpoint 挂起恢复——审批可能隔几小时甚至几天，工作流需要持久化中间状态。
> 
> **检索与上下文**：检索用 SQLite FTS5 加向量检索双路融合，用 RRF 排序，k=60。检索结果不直接喂模型，中间有一个独立的 context assembly 层，做去重、token 预算裁剪和安全过滤。这样检索和模型调用是解耦的，我切检索方案不影响下游。
> 
> **工具调用安全**：我没有做开放式工具调用。工具只有 3 个，全部 read-only，通过白名单注册表管理。调用前有 workflow 维度的权限隔离和 JSON Schema 运行时校验，调用后有 guardrail 拦截。模型的 tool call 不安全时直接降级到只返回检索结果。
> 
> **写回安全**：Agent 对笔记的修改必须经过完整的审批链路。后端生成结构化 proposal，插件端展示 diff 预览，用户审批后执行受限的 patch 操作，写回前后有 hash 校验，所有操作写 append-only 审计日志。
> 
> **评测**：我搭了一个离线评测框架，21 条 golden case 分成问答和治理两组 benchmark，有确定性断言。后端还有 104 条测试。整个项目我独立完成。

### 5 分钟版本

> 我做的是一个基于 LLM 的个人知识治理 Agent 系统。先说一下业务背景：我用 Obsidian 做学习笔记，积累了不少 Markdown 文档，但整理、复盘、打标签这些事纯靠手工太低效。市面上的方案要么是纯 chatbot 只能问答不能治理，要么是让 AI 直接改文件用户完全失控。我想要的是"AI 负责分析和建议，人负责审批和决策"。
> 
> **架构选型**：我选了 workflow-first 架构，有三条工作流：ask 负责问答、ingest 负责文档治理、digest 负责周期复盘。没有用 ReAct 的原因是治理场景需要确定性——比如 ingest 的链路是"检索相关内容 → 分析治理需求 → 生成结构化 proposal → checkpoint 挂起等审批 → 用户批准后受控写回"。这个流程必须是固定的，不能让模型自己决定。
> 
> 编排层我自研了一个 checkpointed linear graph runner。它兼容 LangGraph 的 StateGraph 接口——当 LangGraph 装了就用原生的，没装就用我写的一个轻量 shim。核心能力是 checkpoint 持久化：每个 graph node 执行完都存 SQLite，审批节点挂起后，下次 resume 时从断点继续。thread_id 贯穿整个生命周期。
> 
> **检索管线**：我用的是 FTS5 加向量双路检索，用 RRF 融合排序，k=60。为什么不单路？FTS 擅长精确关键词但不懂语义，向量擅长语义但精确匹配弱，双路互补。filter fallback 我放在 hybrid 总入口统一处理，不让两条分支各自放宽，避免条件漂移导致融合结果不一致。
> 
> 检索结果不直接送模型，中间有一个 context assembly 控制面。它做三件事：按 chunk_id 去重、按 token 预算裁剪（当前 2400）、安全标记检测。最后打包成一个 ContextBundle，附带 assembly_notes 记录处理过程，方便调试。
> 
> **工具调用**：我没有做开放式 function calling。工具注册表里只有 3 个工具——search_notes、load_note_excerpt、list_pending_approvals，全部 read-only。调用链路是：模型判断要不要用工具 → guardrail 前置校验（白名单 + workflow scope + read-only + schema 校验）→ 安全才执行 → 执行结果的 safety flag 检查 → 最后 guardrail 后置校验。任何一环不通过就降级到 retrieval_only。
> 
> **写回安全**：这是这个项目的核心差异点。后端生成的 proposal 包含 patch_op 列表、evidence、risk_level 和 safety_checks。插件端拿到 proposal 后展示 diff 预览，用户可以填 reviewer 和 comment。写回只支持两类操作：merge_frontmatter 和 insert_under_heading——有意限制操作类型来降低出错概率。写回前校验 before_hash，写回后生成 after_hash，所有操作写 append-only 审计日志。
> 
> **降级策略**：主链路有多层 fallback。检索层 filter 无结果会自动去 filter 重试。模型层 cloud provider 挂了自动切 local。模型返回的引用编号如果跟候选集不对齐，自动降级为只返回检索片段。guardrail 拦截也是降级而不是报错。所有降级路径统一走 `_build_retrieval_only_result()`。
> 
> **可观测性**：我做了 JSONL + SQLite 双写 trace。关键设计是 compose_trace_hooks——多个 hook 组合在一起，每个独立 try/catch，一个写 JSONL 失败不影响 SQLite 那路。trace_id 用内容的 SHA1 生成，SQLite 端用 ON CONFLICT DO NOTHING 去重。还有 CLI 可以按 thread_id 查完整链路。
> 
> **评测**：21 条 golden case 分成 question_answering 和 governance 两组 benchmark。拆两组是为了面试能说清楚——"问答链路退化了"和"治理链路退化了"是两个不同的故事，不能只有一个全局平均分。断言是确定性的：检查 mode、citations、proposal 结构、audit_log 写入等。
> 
> 整个项目从架构设计到代码实现、评测框架搭建都是我独立完成的。目前可本地运行，有离线评测覆盖，尚未上线生产环境。
