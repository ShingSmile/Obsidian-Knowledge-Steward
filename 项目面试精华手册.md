# 项目面试精华手册

> 目标：只保留后续快速复习和面试表达最有用的内容。

## 1. 项目一句话定位

- [代码事实] 这是一个面向 Obsidian 的双端知识治理系统：插件负责 UI、审批和本地写回，FastAPI 后端负责 LangGraph 工作流、索引、检索、审计、checkpoint 和评估。
- [文档结论] 项目的核心目标不是“做一个能聊天的 Obsidian AI”，而是“持续治理个人学习知识库”。
- [合理推测] 面试时最好把它讲成“知识治理工作流系统”，而不是“RAG 问答插件”，这样 LangGraph、HITL、checkpoint、audit 才有必要性。

## 2. 业务背景与痛点

- [文档结论] 个人学习型 Vault 持续写入后，常见问题不是“不能问答”，而是重复、冲突、孤立、开放问题堆积、复盘断裂和结构腐化。
- [代码事实] 当前系统已经实现三类入口：`ask_qa`、`ingest_steward`、`daily_digest`。
- [合理推测] 这三类入口分别对应“问答兜底”“新/改单笔记治理”“周期复盘”三条主叙事，面试时不要只盯着 ask。

## 3. 项目整体架构

- [代码事实] 输入层：Obsidian 插件提供侧边栏、审批面板、待审批收件箱、后端探活和本地启动控制。
- [代码事实] 编排层：后端通过统一的 `/workflows/invoke` 入口分发到 `ask_graph`、`ingest_graph`、`digest_graph`，三条 graph 共用一套 runtime contract。
- [代码事实] 知识处理层：Markdown parser 会抽取 `note_type`、`template_family`、`daily_note_date`、heading、wikilink、task_count，并写入 SQLite `note/chunk` 索引。
- [代码事实] 检索层：同时支持 SQLite FTS、`chunk_embedding` 向量检索和 RRF hybrid retrieval；ask 与 ingest governance 共用 hybrid candidate 底座。
- [代码事实] 输出控制层：审批类 workflow 会生成 `proposal`、`patch_op`、`proposal_evidence`，并通过插件执行受限写回，再经 `/workflows/resume` 记账。
- [代码事实] 观测与回归层：运行时 trace 会双写 JSONL 和 SQLite `run_trace`；SQLite 中还持久化 `workflow_checkpoint`、`audit_log`、pending proposal；`eval/run_eval.py` 提供离线回归。
- [合理推测] 这套架构的本质是“插件掌握文件修改权限，后端掌握流程状态和可观测性”，两边分工清晰。

## 4. 主执行链路

### 大白话版

- [代码事实] 用户从插件或 API 发起 ask / ingest / digest，请求先进入后端统一入口 `/workflows/invoke`。
- [代码事实] 后端先生成 `thread_id` 和 `run_id`，再进入对应 graph；每条 graph 都会记录 trace，并按需要写 checkpoint。
- [代码事实] ask 会走 hybrid retrieval，再尝试生成带引用答案；如果模型不可用或引用编号不合法，就降级成 retrieval-only。
- [代码事实] ingest 和 digest 在需要审批时，会先生成 `proposal`，再把 proposal 和 `waiting_for_approval` checkpoint 原子写入 SQLite。
- [代码事实] 插件审批面板可以从 `GET /workflows/pending-approvals` 拉取真实待审批项。
- [代码事实] 用户 approve 后，插件本地执行受限写回，校验 `before_hash`，生成 `writeback_result` 回传 `/workflows/resume`。
- [代码事实] 后端会把审批和写回结果写入 append-only `audit_log`，并在成功写回后 best-effort 触发 scoped ingest 刷新索引。
- [代码事实] 如果还是当前插件会话，本地还能基于写回前抓取的快照执行最小 rollback，并再把 rollback 结果回传后端记账。

### 技术版

- [代码事实] 控制面入口在 `backend/app/main.py`，通过 handler registry 把 `WorkflowAction` 映射到 ask / ingest / digest 三类执行器。
- [代码事实] 共享 runtime 在 `backend/app/graphs/runtime.py`，统一了基础 state、trace hook、checkpoint resume 和 `WorkflowGraphExecution`。
- [代码事实] ask 主链路是 `prepare_ask -> execute_ask -> finalize_ask`，核心业务在 `backend/app/services/ask.py`。
- [代码事实] ingest 主链路是 `prepare_ingest -> execute_ingest -> finalize_ingest`；若 `approval_mode=proposal`，会在 execute 后额外生成 retrieval-backed proposal，并把 thread 推进到 waiting。
- [代码事实] digest 主链路是 `prepare_digest -> build_digest -> finalize_digest`；当前 digest 仍是模板化摘要，但审批链路已真实接通。

### 1 分钟面试表达版

- [面试表达] 这个项目的主链路不是“插件调一次模型然后返回文本”，而是“插件发起工作流，后端统一分配 thread_id/run_id，用 LangGraph 管理状态和 checkpoint；如果只是 ask，就直接返回带 citation 的答案；如果涉及 digest 或治理写回，就先生成 proposal，原子落 proposal 和 waiting checkpoint，再由插件承接人工审批和本地受限写回，最后把 writeback_result 回传后端做 audit、checkpoint 更新和 scoped reindex”。这样我能把文件修改权限、安全审查、流程恢复和评估回归拆开做，而不是全压在一次 LLM 调用里。

### `ask_qa` 链路逐段拆解

- [代码事实] 请求先进入 `WorkflowInvokeRequest`，关键入参是 `user_query`、`retrieval_filter`、`provider_preference`；`backend/app/main.py` 负责生成 `thread_id/run_id` 并路由到 `invoke_ask_graph()`。
- [代码事实] `ask_graph` 的 `prepare_ask` 节点只把 `query_length`、filter 摘要和 provider 选择写入 trace，不直接把原始 query 文本全量抄进观测数据。
- [代码事实] `execute_ask` 会调用 `run_minimal_ask()`：先做 hybrid retrieval，拿到 top-k candidate；如果没有命中，直接返回 `NO_HITS`。
- [代码事实] 命中候选后，系统会按“首选 provider -> 备选 provider”的顺序尝试生成 grounded answer；默认策略是云优先、本地兼容。
- [代码事实] 模型生成后不会被直接信任，还会做程序级 citation 对齐校验：没有 `[n]`、编号越界，都会自动降级成 `retrieval_only`。
- [为什么这样设计] ask 链路的核心不是“尽量生成一段看起来聪明的话”，而是“先把证据集合稳定住，再决定能不能安全生成答案”。
- [面试表达] 这条链路可以概括成“先证据、后生成、再校验”。模型只是证据的组织器，不是最终真相来源。

### `INGEST_STEWARD -> approval -> resume` 链路逐段拆解

- [代码事实] ingest 入口会先把 `note_path/note_paths` 规范化成稳定的 `note_paths` 列表，checkpoint 恢复时也按这个字段匹配，避免同一 scope 因参数形态不同而串状态。
- [代码事实] `execute_ingest` 先真实执行 scoped/full-vault ingest，把 note/chunk 和必要的 embedding 写入索引；审批 proposal 不是“只做分析不落索引”，而是建立在真实 ingest 之后。
- [代码事实] 只有显式 `approval_mode=proposal` 且当前是单 note scoped ingest 时，才会进入 retrieval-backed proposal 路径；多 note 直接跳过 proposal。
- [代码事实] proposal 生成后，会把 `proposal` 和 `waiting_for_approval checkpoint` 放在同一个 SQLite 事务里原子提交，`next_node_name` 明确写成 `human_approval`。
- [代码事实] 插件侧从 pending inbox 读取 proposal，用户点击 approve 时，本地先执行 `applyProposalWriteback()`；如果本地已改成功但后端 `/workflows/resume` 失败，插件会保留 `pendingWritebackExecution`，允许只重试后端记账，不重复改文件。
- [代码事实] `/workflows/resume` 会严格校验审批和写回结果：reject 不能带 `writeback_result`，approve 暂不支持 partial patch，已应用的 patch 集合必须和 proposal 完全一致，`before_hash` 也要对齐 proposal 的安全检查。
- [代码事实] resume 成功时，append-only `audit_log` 和 completed checkpoint 也在同一个事务里提交；如果写回成功，还会 best-effort 触发 scoped ingest，并通过 `post_writeback_sync` 返回索引刷新结果。
- [代码事实] rollback 不是任意时刻都能做，只有当前 completed checkpoint 里已经有 successful writeback result，且插件本地还能证明文件仍命中精确 `after_hash` 时，才允许保守恢复。
- [为什么这样设计] 这条链路的核心思想是把 side effect plane 和 control plane 拆开：插件负责真实改文件，后端负责状态一致性、审计和恢复协议。
- [面试表达] 这条链路最值得讲的不是“我能自动改笔记”，而是“我怎么避免改错、怎么保证 proposal 不丢、怎么在本地写回和后端记账之间做最小补偿”。

## 5. 核心模块拆解

### 5.1 插件层

- [代码事实] 关键文件：`plugin/src/main.ts`、`plugin/src/views/KnowledgeStewardView.ts`、`plugin/src/backend/runtime.ts`、`plugin/src/writeback/applyProposalWriteback.ts`。
- [代码事实] 插件负责打开面板、拉 pending inbox、展示 proposal、发起 approve/reject、执行本地写回和本地 rollback。
- [为什么这样设计] 写回必须放在插件侧，因为真正掌握 Obsidian Vault API 和宿主上下文的是插件，不是本地 FastAPI。
- [关键实现点] 先校验 `before_hash`，只支持受限 `patch_op`，approve 后本地改文件，再把结果回传后端统一落审计。
- [风险点] rollback 目前只支持“当前插件会话里、且文件仍命中精确 after_hash”的保守恢复。

### 5.2 后端控制面

- [代码事实] 关键文件：`backend/app/main.py`、`backend/app/contracts/workflow.py`。
- [代码事实] 后端对外暴露 `/health`、`/workflows/invoke`、`/workflows/resume`、`/workflows/rollback`、`/workflows/pending-approvals`。
- [为什么这样设计] 先统一 workflow contract，再把具体业务挂进去，避免 ask / ingest / digest 各自复制入口样板。
- [关键实现点] `WorkflowInvokeRequest/Response`、`WorkflowResumeResponse`、`WorkflowRollbackResponse` 把线程状态、proposal、writeback_result、post_writeback_sync 统一成同一套协议。
- [风险点] 现在是“三条 workflow 共用控制面”，不是“任意 workflow 可插拔的平台”。

### 5.3 Graph Runtime 与恢复机制

- [代码事实] 关键文件：`backend/app/graphs/runtime.py`、`backend/app/graphs/checkpoint.py`、`backend/app/graphs/state.py`。
- [代码事实] runtime 会在每个 step 后写 `workflow_checkpoint`，支持 `resume_from_checkpoint`，并在 completed checkpoint 且终态完整时短路返回。
- [为什么这样设计] 这能支撑 approval 前等待、approval 后恢复、失败后排障，以及 interview 时最重要的“可中断、可恢复、可审计”叙事。
- [关键实现点] 恢复时只复用业务状态，不复用旧 `run_id` 和旧 trace；trace 是新 run，业务状态是旧 thread。
- [风险点] 目前 waiting approval 的恢复走 `/workflows/resume`，不是通用状态机平台。

### 5.4 索引与检索层

- [代码事实] 关键文件：`backend/app/indexing/parser.py`、`backend/app/indexing/ingest.py`、`backend/app/indexing/store.py`、`backend/app/retrieval/sqlite_fts.py`、`backend/app/retrieval/sqlite_vector.py`、`backend/app/retrieval/hybrid.py`。
- [代码事实] parser 会把 Markdown 解析成 note/chunk，并抽出标题、heading path、wikilink、task 数、模板类型和日期。
- [代码事实] SQLite 里已经有 `note`、`chunk`、`chunk_fts`、`chunk_embedding`、`workflow_checkpoint`、`proposal`、`proposal_evidence`、`patch_op`、`audit_log` 等表。
- [为什么这样设计] 单机单用户场景优先选择 SQLite，既能做结构化存储，也能做 FTS、审计和 checkpoint，部署成本最低。
- [关键实现点] hybrid retrieval 用 RRF 融合 FTS 和向量；metadata filter fallback 在 hybrid 总入口统一处理，避免两条分支条件漂移。
- [风险点] 向量分支不可用时会稳定降级到 FTS，但向量覆盖率观测和更细粒度增量 FTS 还没做完。

### 5.5 治理与审批恢复

- [代码事实] 关键文件：`backend/app/services/ingest_proposal.py`、`backend/app/services/resume_workflow.py`、`backend/app/services/rollback_workflow.py`。
- [代码事实] ingest proposal 当前只覆盖“单 note scoped ingest”，会结合 related retrieve 结果给出最小治理建议，并显式排除 target note 自证。
- [为什么这样设计] 先把最小可信治理闭环做通，再考虑多 note 冲突裁决，否则范围会迅速失控。
- [关键实现点] proposal 与 waiting checkpoint 在同一事务里原子写库；resume 会校验审批结果和 writeback_result 的一致性，再落 audit 和 post-writeback sync。
- [风险点] 现在的治理问题类型仍偏保守规则驱动，主要是 `orphan_hint`、`duplicate_hint` 这类最小范围。

## 6. 技术选型与权衡

- [代码事实] 选择 LangGraph 的直接理由不是“多 agent 很炫”，而是当前链路确实需要 checkpoint、thread_id、waiting_for_approval 和恢复协议。
- [代码事实] 当前模型配置是“云优先、本地兼容”，`KS_DEFAULT_MODEL_PROVIDER` 默认是 `cloud`；这和初步实现指南里的“本地优先”已经发生偏移。
- [代码事实] 选择插件 + 本地后端双端架构，是因为插件适合 Vault IO 和审批 UI，后端适合索引、检索、图编排、评估和持久化。
- [代码事实] 选择 SQLite-first，是因为这个项目首先是个人知识库场景，重点是可演示、可回放、可审计，而不是分布式吞吐。
- [代码事实] 选择 hybrid retrieval，是因为治理和问答都既需要关键词精确命中，也需要语义召回；单独 FTS 或单独向量都不够稳。
- [合理推测] 面试时可以把它定义为“workflow-first agent application”：有 agent 味道，但工程落地形态更接近状态机工作流，而不是 ReAct 自由代理。

## 7. 可讲亮点

- [代码事实] 亮点 1：三条 workflow 共用统一控制面。
  面试说法：我不是分别写了三个 API 分支，而是先统一了 thread/run、trace、checkpoint、response builder，再把 ask / ingest / digest 作为不同业务图挂进去。
- [代码事实] 亮点 2：proposal 与 waiting checkpoint 原子落库。
  面试说法：我重点解决的是“插件拿到了 proposal，但后端没进入 waiting”这种状态错位问题，所以 proposal 和 checkpoint 必须同事务提交。
- [代码事实] 亮点 3：写回走插件本地执行，后端只做恢复和审计。
  面试说法：这样可以把文件修改权限收束在宿主内，用 `before_hash`、受限 patch op、audit_log 和 rollback 把风险压住。
- [代码事实] 亮点 4：治理和问答共用 hybrid retrieval 底座。
  面试说法：我没有把 retrieval 只当问答底层，而是让治理 proposal 也消费同一套证据候选，这样知识治理不是纯拍脑袋规则。
- [代码事实] 亮点 5：项目自带离线 eval 和场景分桶 benchmark。
  面试说法：我不是只演示 happy path，而是把 ask / governance / digest / resume-writeback 分场景做成可重复回归。

## 8. 不要乱吹的点

- [代码事实] 不要说这是“多 agent 协作平台”。当前是三条 workflow graph，共享 runtime contract，不是 supervisor 或 agent swarm。
- [代码事实] 不要说 digest 已经做了真实 LLM 聚类、规划和调度。当前 digest 还是模板化摘要。
- [代码事实] 不要说 ask 已经有运行时语义级 groundedness gate。当前做到的是 citation 编号对齐校验，语义级风险主要还停留在离线 eval。
- [代码事实] 不要说 rollback 支持跨会话、历史写回或智能三方合并。当前 rollback 边界非常保守，只覆盖当前插件会话内的精确恢复。
- [代码事实] 不要说 ingest proposal 已经支持多 note 冲突治理。当前只支持单 note scoped ingest proposal。
- [文档结论] 不要继续按初步实现指南把模型路线讲成“本地优先”；当前事实已经收敛为“云优先、本地兼容”。

## 9. 高频追问题库

- [问题] 你为什么一定要用 LangGraph，而不是 FastAPI 里写几个函数串起来？
  [参考回答] 因为我的核心价值不只是调模型，而是需要 thread 级状态、checkpoint、waiting_for_approval 和恢复协议。普通函数链可以跑通 happy path，但很难把中断恢复、审计和 replay 讲清楚。
- [问题] 为什么写回不放后端做？
  [参考回答] 因为真正控制 Vault 的是 Obsidian 宿主，插件天然拥有文件 API、metadata cache 和用户交互面板。后端负责状态和审计，插件负责最终落盘，更安全也更符合权限边界。
- [问题] 你这个项目到底是 agent，还是 workflow？
  [参考回答] 我会说它是 workflow-first 的 agent 应用。因为它有检索、规划式决策和工具调用，但落地形态更强调确定性状态流、恢复和审批，而不是开放式自主代理。
- [问题] 为什么治理 proposal 也要接 retrieval？
  [参考回答] 如果只看当前 note，本质上是规则补丁；接 retrieval 后才能把“这个笔记和已有哪些旧笔记重复/冲突/相关”作为证据链带出来。
- [问题] 你怎么保证 proposal 不会丢，resume 不会串？
  [参考回答] proposal 和 waiting checkpoint 是同事务落库；resume 还会校验 thread_id、proposal_id、approval_decision 和 writeback_result 是否匹配当前 checkpoint。
- [问题] 如果模型生成了带错编号的引用怎么办？
  [参考回答] ask 服务端会做程序级 citation 对齐校验，缺编号或越界就直接降级成 retrieval-only，不让“像真的一样”的坏答案继续往下游走。
- [问题] 你现在最大的边界是什么？
  [参考回答] digest 仍偏模板化；ask 还没有运行时语义 groundedness gate；rollback 只支持当前插件会话内的保守恢复；这些我都会主动说明，不会硬吹。

## 10. 面试口语稿

### 30 秒版

- [面试表达] 我做的是一个基于 LangGraph 的 Obsidian 知识治理系统，不是普通聊天插件。它把 ask、笔记治理和 daily digest 都统一成可恢复的 workflow；涉及写回时先生成 proposal，人工审批后由插件本地执行受限写回，再把结果回传后端做 audit、checkpoint 和索引刷新，所以重点是可控、可审计、可评估。

### 1 分钟版

- [面试表达] 这个项目解决的是个人知识库越写越乱的问题，而不是单纯做问答。我把系统拆成插件和本地后端两层：插件负责 UI、待审批收件箱和本地写回，后端负责 LangGraph 工作流、SQLite 索引、hybrid retrieval、checkpoint 和 eval。统一入口是 `/workflows/invoke`，支持 ask、ingest、digest 三类 action。ask 走 hybrid retrieval 和 citation 校验；ingest 和 digest 在需要改笔记时会先生成 proposal，把 proposal 和 waiting checkpoint 原子写库，再由插件审批和执行写回，最后通过 `/workflows/resume` 统一落审计和 post-writeback scoped ingest。这样我能把知识治理、人工在环和可观测性都讲清楚。

### 3 分钟版

- [面试表达] 这个项目的背景是，个人学习型 Obsidian Vault 里最难的问题通常不是“我能不能问一句话”，而是内容持续写入后会出现重复、冲突、孤立和复盘断裂。所以我没有把它做成聊天助手，而是做成一个 workflow-first 的知识治理系统。整体上是插件加本地后端双端架构：插件负责面板、待审批收件箱、后端探活和本地受限写回；后端负责 FastAPI 控制面、LangGraph 编排、SQLite 索引、hybrid retrieval、checkpoint、audit 和离线 eval。

- [面试表达] 具体链路上，用户先通过插件或 API 发起 ask、ingest 或 daily digest。后端统一分配 thread_id 和 run_id，再进入对应 graph。ask 会先做 FTS + 向量的 hybrid retrieval，如果模型能生成合法 citation 的答案就返回 grounded answer，否则就降级成 retrieval-only。ingest 和 digest 如果涉及写回，不会直接改文件，而是先生成 proposal，并把 proposal 和 waiting_for_approval checkpoint 原子写到 SQLite。插件审批面板会从 pending inbox 里拉真实 proposal，用户 approve 后，插件本地校验 before_hash，只执行受限 patch op，再把 writeback_result 回传 `/workflows/resume`。后端收到后会校验审批和写回结果是否一致，写 append-only audit_log，必要时做 scoped ingest 刷新；如果还是当前插件会话，还支持一次保守 rollback。

- [面试表达] 我觉得这个项目最有价值的地方有三个。第一，它把知识治理做成了真正可恢复、可审计的工作流，而不是一次 prompt。第二，它把写回权限收束在插件宿主内，用 before_hash、proposal、audit 和 rollback 控制风险。第三，它不是只做 demo，还补了按 ask / governance / digest / resume-writeback 分场景的离线 eval。边界我也会主动说清楚，比如 digest 现在还是模板化，ask 只做到了 citation 编号级校验，还没上运行时语义 groundedness gate，rollback 也只支持当前插件会话内的保守恢复。
